## Context

Wiki 需要**自动跟随游戏更新**——类似 CI/CD，当 Rimworld 发布新版本时，pipeline 自动从 Steam CDN 拉取最新数据并重建 wiki。

核心问题：有没有**不需要账号密码**的办法？

## 调研结论：没有免费方案

### 为什么必须认证

Rimworld (App ID 294100) 是**付费游戏**（¥128），`is_free: false`。

Steam CDN 的认证规则：

| 游戏类型 | 匿名访问 | 示例 |
|---------|---------|------|
| Free to Play | 可以 | Dota 2, CS2, TF2 |
| 付费游戏 | **不行** | Rimworld, Factorio |
| 有 Dedicated Server 的游戏 | Server 部分可以 | 某些多人游戏 |

Rimworld 没有独立的 Dedicated Server depot，所以**所有内容都需要认证**。

### 已排除的方案

| 方案 | 为什么不行 |
|------|----------|
| DepotDownloader 匿名模式 | 付费游戏不支持 |
| SteamCMD anonymous | 同上 |
| ValvePython/steam 匿名 | 同上 |
| Family Share + DepotDownloader | 见下方详细分析 |
| Ludeon 官方 API | 不存在 |
| rimworldwiki.com 爬取 | 返回 403，有反爬 |
| GitHub 社区 XML 仓库 | 没有找到维护 Defs 原始 XML 的仓库 |
| Rimworld-Auto-Documentation | 是生成的 HTML，不是原始 XML |
| Steam Web API (appdetails) | 只有商店信息，没有游戏文件 |

### Family Share 详细分析

思路：用你已有的账号开启 Family Sharing，授权一个新账号（免费创建）共享 Rimworld，然后用新账号登录 DepotDownloader 下载。

**结论：不可行。** 原因：

1. **DepotDownloader 不支持 Family Share**
   - GitHub Issue [#447](https://github.com/SteamRE/DepotDownloader/issues/447) 专门问了这个问题
   - 已关闭，无实现计划
   - Steam 的 PICS/CDN 协议在 depot 层面检查 ownership，Family Share 授权发生在 Steam 客户端层面，不传递到 CDN 协议

2. **Family Share 是按设备授权的**
   - 授权绑定到具体电脑的 Steam 客户端实例
   - GitHub Actions runner 是临时机器，每次运行都是新机器
   - 无法在 CI 环境中完成设备授权流程

3. **并发限制**
   - 同一时间只能有一个用户访问共享库
   - 如果你在本地玩 Rimworld，CI 同时下载会冲突

4. **所有者优先**
   - 游戏所有者随时可以中断共享访问
   - CI 运行的可靠性无法保证

### 唯一可行方案：Steam 账号 + DepotDownloader

付费游戏必须通过认证下载，这是 Steam 的硬限制，没有绕过办法。

## Decision

**使用 DepotDownloader + Steam 账号，但通过以下措施降低风险：**

### 1. 专用 Steam 账号

- 创建一个**专门用于 CI 的 Steam 账号**（不用个人账号）
- 购买 Rimworld（一次性 ¥128）
- 启用 Steam Guard，生成**专用密码**（不是主密码）

### 2. GitHub Secrets 管理

```yaml
# 存储在 GitHub Secrets 中，不在代码中出现
env:
  STEAM_USER: ${{ secrets.STEAM_CI_USER }}
  STEAM_PASS: ${{ secrets.STEAM_CI_PASS }}
```

- Secrets 加密存储，日志中自动脱敏
- 只有 GitHub Actions 运行时可访问
- 可随时轮换密码

### 3. Session 缓存减少 2FA 交互

DepotDownloader 支持 `-remember-password`，保存 session key 到本地。可以用 GitHub Actions cache 持久化：

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.local/share/DepotDownloader
    key: steam-session-${{ hashFiles('filter.txt') }}
```

这样大部分 CI 运行不需要重新输入 2FA。

### 4. 最小权限原则

- 只下载 `Defs/*.xml`（12MB），不下载完整游戏
- 使用 `-filelist filter.txt` 精确过滤
- CI 运行频率低（每周一次或手动触发）

### 5. 触发策略

```yaml
on:
  schedule:
    - cron: '0 8 * * 1'    # 每周一检查
  workflow_dispatch:         # 手动触发（游戏更新时）
```

检测更新的逻辑：
1. 下载 manifest only（`-manifest-only`）
2. 对比上次 manifest ID
3. 有变化才下载实际文件

## Consequences

优点：

- 真正的 CI/CD 自动更新
- DepotDownloader 成熟稳定，维护成本低
- GitHub Secrets 安全管理凭据
- Session 缓存减少人工干预

缺点：

- 需要一次性购买游戏（¥128）
- 需要维护一个 Steam 账号
- 理论上 Valve 可以封锁第三方工具（极低概率）
- 2FA 可能在密码过期时需要人工干预

## 安全风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| Steam 账号被封 | 低 | 专用账号，不用主账号 |
| 凭据泄露 | 低 | GitHub Secrets 加密 |
| DepotDownloader 被封 | 极低 | 备选 SteamCMD |
| 2FA 阻断 CI | 中 | Session 缓存 + 专用密码 |

## 结论

**没有不需要账号的办法。** 这是 Steam 付费游戏的硬限制。

- 匿名访问：付费游戏不支持
- Family Share：DepotDownloader 不支持，且 CI 环境无法完成设备授权
- 官方 API：不存在

最小代价是 ¥128 买一个专用账号 + GitHub Secrets 管理凭据。
